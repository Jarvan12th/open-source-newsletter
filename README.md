# 开源新闻信息雷达

这是一个面向开源领域的新闻工作仓库，目标不是单纯保存链接，而是持续捕获值得追踪、值得研判、能转化为内容选题的重要开源动态。

这个仓库沿用了 `tech-newsletter` 的轻量工作流，但把监测对象切换为开源生态，重点聚焦六类信号：

- 重要项目发布、版本更新与路线图变化
- 基金会、标准组织、托管平台和治理结构调整
- 安全事件、许可证变化、维护状态和依赖风险
- 明星项目、开发工具和基础设施的生态扩散
- GitHub 等平台上突然走热、容易理解、容易上手实践的开源项目
- 维护者、社区、资金支持和商业化相关动态

## 各层职责

### `00_收集箱`

只负责先接住线索，不在这里做最终判断。

- `待整理链接`：看到但还没核实的项目、文章、issue、release note
- `快速记录`：判断、灵感、潜在标题、风险提醒
- `截图与素材`：GitHub 页面、趋势榜、发布图、讨论截图

### `01_信息雷达`

这是核心层，负责稳定捕获开源领域的重要变化。

- `01_每日扫描`：按巡检日期沉淀每日候选池
- `02_重点信源`：维护长期监测入口和固定观察面
- `03_热点追踪`：给持续发酵的项目或事件建单独条目
- `04_候选选题`：把新闻线索转成可做内容的主题
- `05_监测规则`：维护关键词、信源注册表、评分标准和方法

### `02_选题研判`

把“看见更新”升级成“形成判断”。

- `热点事件`：适合短周期快速响应的项目或争议
- `长期方向`：AI 开源、云原生、数据库、开发者工具、安全等长期主线
- `项目生态`：重点项目、基金会、公司与维护者生态关系
- `周度综述`：输出每周热点脉络摘要，直接服务内容选题

### `03_内容生产`

只保留与输出内容直接相关的目录。

- `选题脚本`：提纲、脚本、角度设计
- `发布文案`：标题、摘要、平台文案
- `封面与配图`：封面图、截图、生态图整理

### `04_发布复盘`

发布后记录成效，反向优化监测方法。

- `已发布`：已上线内容归档
- `数据复盘`：点击、讨论度、收藏、评论反馈

## 开源新闻的使用原则

1. 优先一手源。尽量回到 release note、官方博客、项目仓库、基金会公告。
2. 区分代码变化和舆论变化。Star 增长不等于真实采用，社区争议也不等于项目衰退。
3. 关注维护状态。项目活跃度、维护者变动、issue 响应、版本节奏都很关键。
4. 把安全和许可证单独看待。这类事件热度未必最高，但影响通常更深。
5. 同一事件要归并。官方发布、媒体报道、社区讨论不要分散记成多条。
6. 允许时间偏差。只要在近 1-3 天内仍有跟进价值，都可以纳入当日巡检。

## 推荐工作节奏

- 每天：更新 `01_信息雷达/01_每日扫描/` 的每日候选池
- 有持续热度时：在 `01_信息雷达/03_热点追踪/` 建事件卡片
- 每周：在 `02_选题研判/周度综述/` 输出每周热点脉络摘要
- 出稿前：从周摘要和候选选题里补齐背景、争议点和行业判断
- 发布后：到 `04_发布复盘/` 记录反馈，反哺关键词和信源注册表

## 四个固定接口

为了方便人工执行和后续自动化，信息雷达层固定为四个接口：

- `source_tracker`：`01_信息雷达/05_监测规则/信源注册表.md`
- `daily_radar_entry`：`01_信息雷达/01_每日扫描/每日候选池模板.md`
- `weekly_radar_digest`：`02_选题研判/周度综述/每周热点脉络摘要模板.md`
- `automation_radar_rules`：`01_信息雷达/05_监测规则/自动化巡检说明.md`

最小约束如下：

- 任一事件必须能追溯到原始来源链接
- 任一周摘要必须能回溯到对应的每日候选条目
- 同一事件多次出现时应归并，而不是重复计数
- 安全、许可证、维护状态变化应优先标注影响范围
- 热门项目不得只按 star 数判断，必须同时看可理解性、可实践性、维护活跃度和讨论质量
